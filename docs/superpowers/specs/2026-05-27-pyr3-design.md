# pyr3 — v1.0 design spec

> Strategic design doc for pyr3 — a TypeScript + WebGPU successor to pyr3-kotlin,
> pyr3-peek, and pyr3-rust. Locks the load-bearing architectural decisions and lays out
> the phase plan to v1.0 ship gate.

## 1. Vision

**One engine, two consumers, both honest about flam3 lineage.**

pyr3 is a fractal-flame renderer in the flam3 family, written in pure TypeScript with WGSL
compute shaders. The same `.ts` modules and the same `.wgsl` shaders drive:

- A **browser viewer** — Vite + WebGPU + GitHub Pages.
- A **headless CLI** — Node + the `webgpu` npm package (`dawn-gpu/node-webgpu`) + `tsx`.

The "single engine" is not a marketing slogan — it's a literal source-tree property: engine
modules have **zero environment branching**. The CLI binary stamps `webgpu` globals onto
`globalThis` (≈8 lines of bootstrap) and calls the same `createRenderer()` the browser does.

## 2. Locked architectural decisions

| # | Decision | Reason |
|---|---|---|
| 1 | TS + WebGPU + Vite | User-given. Matches pyr3-peek's proven stack. |
| 2 | Node + `webgpu` npm (`dawn-gpu/node-webgpu`) | Dueling-agent research (3 parallel agents, 2026-05-27). Same Dawn as Chromium → best shot at cross-consumer equivalence. 33K weekly dl, darwin-universal prebuilt, active. Beat Deno (`--unstable-webgpu` flag, spec stuck at March 2024 draft, active regressions), Bun (`bun-webgpu` 0.1.7 with known FFI leak in `buffer.unmap()`), and `@kmamal/gpu` (9 months stale). |
| 3 | Vitest + tsx | Proven in pyr3-peek; matches the runtime; no harness rewrite. |
| 4 | GPU only, no CPU path | pyr3-kotlin learned: CPU is ~50× slower than GPU on same hardware; "match flam3 C-speed" is unwinnable. |
| 5 | Frontend = pyr3-peek layout for v1.0 | User-specified. Editor is much-later post-v1 (`[PYR3-001]`). |
| 6 | "similar but not the same" as flam3-C | Kotlin contract, ported. |
| 7 | Share links = peek's `?flame=<inline-encoded>` URL | Permanent, no remote dependency. |
| 8 | Phase plan: 0 (copy) → 1 (kotlin audit-port) → 2 (test rig) → 3 (iterate to ship gate) | User-locked execution order. Phase 1 BEFORE ship-gate work so bug debt is paid upfront. |

## 3. The v1.0 ship gate

> **Both pyr3-FE-render AND pyr3-BE-render match flam3-C-render within R tolerance, on the
> chosen fixture set.** FE and BE do not need to match each other byte-for-byte — they each
> independently pass R-vs-flam3.

This is the single sentence the project hangs on. It is the trigger to:

- Cut **v1.0**
- Replace `github.com/MattAltermatt/pyr3` on GitHub (archive the kotlin one)
- Archive `github.com/MattAltermatt/pyr3-peek` on GitHub
- Re-point any share-link URLs

The R metric is ported verbatim from pyr3-kotlin's
`parity/src/main/kotlin/pyr3/parity/Compare.kt`. R-tolerance threshold (likely ~1.5–3.0 based
on kotlin's experience) is calibrated during **Phase 2** against the fixture set.

## 4. Architecture

### 4.1 The engine surface

After Phase 0 (copy peek), the source tree looks like:

```text
pyr3/
├── src/
│   ├── genome.ts            # Genome type + parsing/serializing infra
│   ├── flame-import.ts      # .flame XML → Genome (uses DOMParser; happy-dom in Node)
│   ├── serialize.ts         # JSON ⇄ Genome
│   ├── loader.ts            # File-type sniffing (.flame vs .pyr3.json)
│   ├── variations.ts        # 99 flam3 variation arms (TS impl + WGSL counterparts)
│   ├── isaac.ts             # ISAAC RNG (TS, for non-rendering paths)
│   ├── palette.ts           # Palette lookup, hue rotation, STEP/LINEAR interpolation
│   ├── calibration.ts       # Camera & sample-density derivation
│   ├── chaos.ts             # Chaos-game host-side glue
│   ├── density.ts           # Density estimator host-side glue
│   ├── render-orchestrator.ts  # Top-level: genome → device → output
│   ├── renderer.ts          # createRenderer(device, format, opts) — THE SEAM
│   ├── main.ts              # Browser entry (Vite)
│   ├── ui-bar.ts            # Browser chrome
│   └── shaders/
│       ├── chaos.wgsl
│       ├── density.wgsl
│       ├── spatial-filter.wgsl
│       ├── visualize_u32.wgsl
│       └── visualize_f32.wgsl
├── bin/
│   ├── pyr3-render.ts            # CLI: render one flame to a PNG
│   ├── pyr3-bench.ts             # CLI: perf bench
│   ├── flame-to-json.ts          # CLI: .flame → JSON dump
│   ├── wgsl-loader-register.mjs  # Node loader hook (registration)
│   └── wgsl-loader.mjs           # Node loader hook (impl: handles Vite's ?raw syntax)
├── tests/                   # Vitest suites (engine units + integration)
├── fixtures/                # Reference .flame files + (Phase 2) flam3-C goldens
├── scripts/                 # Diff/dump utilities
├── public/                  # Static assets for Vite
├── index.html               # Vite entry
├── package.json
├── vite.config.ts
├── vitest.config.ts
├── tsconfig.json
└── ...docs (this file et al.)
```

`createRenderer(device, format, opts)` is the seam. Returns an object with a
`render({ genome, outputView, ...flags })` method. Both consumers call this with no knowledge
of which environment they're in.

### 4.2 Frontend consumer (browser)

`src/main.ts` + `index.html` + Vite. WebGPU adapter acquired from `navigator.gpu`. Output
goes to a `<canvas>` via `device.createTexture()` → render → swap-chain. Same
`renderer.render()` call as the CLI.

`?flame=<inline-encoded>` URL parameter loads the flame from the query string at boot — no
remote fetch, no 404 risk. Permanent share links.

### 4.3 Backend consumer (Node CLI)

`bin/pyr3-render.ts` (per pyr3-peek's pattern). ~8-line bootstrap:

```ts
import { create, globals } from 'webgpu';
import { Window } from 'happy-dom';
Object.assign(globalThis, globals);                              // GPUBufferUsage etc.
(globalThis as any).DOMParser = new Window().DOMParser;           // for .flame XML
const navigator = { gpu: create([]) };
```

Then identical-to-browser engine code runs:

```ts
const adapter = await navigator.gpu.requestAdapter();
const device = await adapter.requestDevice({ requiredLimits: { ... } });
const renderer = createRenderer(device, 'rgba8unorm',
  { width, height, oversample, filterRadius });
const texture = device.createTexture({ ... });
renderer.render({ genome, outputView: texture.createView() });
// copyTextureToBuffer → mapAsync → PNG.sync.write → writeFileSync
```

Output: `out.png`. Same logic, different sink.

`wgsl-loader.mjs` provides Node loader-hook support for Vite's `import shader from
'./x.wgsl?raw'` syntax so engine modules import shaders identically in both consumers.

### 4.4 Test harness

Vitest. Both consumers tested in Node:

- **Engine unit tests** — TS modules tested directly against fixtures (variations, RNG,
  palette, calibration, flame-import, serialize).
- **BE render parity** — `pyr3-render.ts` invoked from Vitest; output R-compared to flam3-C
  goldens.
- **FE render parity** — `chrome-devtools-mcp` drives the dev server; captures the canvas;
  output R-compared.

Both parity tests gate v1.0.

## 5. Phase plan to v1.0

### Phase 0 — Copy pyr3-peek wholesale, lay groundwork (target: v0.1)

Mechanical. Steps:

1. `cp -R pyr3-peek/{src,bin,scripts,tests,public,fixtures,index.html,vite.config.ts,vitest.config.ts,tsconfig.json,package.json}` into `pyr3/`.
2. Edit `package.json` — change `name`, `description`, leave version at `0.1.0`, keep
   dependencies.
3. Strip peek-specific scope comments in `src/`, `bin/` headers; keep peek attribution in
   `NOTICE.md` (already seeded).
4. `npm install`.
5. `npm test` — verify ported tests pass.
6. `npm run dev` — open the browser viewer in Chrome, verify a flame renders.
7. `npm run render <fixture> out.png` — verify the CLI produces a PNG of the same flame.
8. Commit as **v0.1**: `engine: Phase 0 — copy pyr3-peek as TS+WGPU basis`.

**Acceptance:** browser viewer renders the welcome flame in Chrome; CLI produces a PNG of
the same flame. Both run end-to-end without errors. Tests green.

### Phase 1 — Audit-port pyr3-kotlin's GPU/parser/variation fixes (target: v0.5)

Enumerate every pyr3-kotlin commit (v0.10 → v1.x-E) that touches `:gpu` (WGSL), `:flam3`
(parser), `:core` (variations / calibration / tonemap / palette). For each: evaluate whether
peek's TS/WGSL has the same bug. If yes, port. Each port is its own commit with a
`Port: pyr3-kotlin <ref>` body trailer citing the kotlin source.

Known-load-bearing ports queued:

| kotlin ref | What | WGSL file |
|---|---|---|
| v0.36-A | EDISC EPS-clamp (near-unit-circle precision crater) | `chaos.wgsl` |
| v0.36-H | sub-ulp walker jitter (fractalapple tight-orbit recovery) | `chaos.wgsl` |
| v1.x-C | finalxform opacity gate (`opacity-=1` short-circuit) | `chaos.wgsl` |
| v1.x-E | post-process pipeline ordering (DE + spatial on readback) | render-orchestrator |
| v0.32 | TonemapPass u32 signedness fix | `visualize_u32.wgsl` |
| v0.28b | DE u32 signedness fix | `density.wgsl` |
| v0.21 | `pre_blur` variation | `variations.ts` + `chaos.wgsl` |

Full enumeration happens during phase execution by scanning kotlin's CHANGELOG.

**Acceptance:** all known kotlin GPU/shader/variation/parser fixes accounted for (ported, or
marked non-applicable with one-line reason in commit body). `npm test` green.

### Phase 2 — Flam3-vs-pyr3 test rig + golden fixture set (target: v0.7)

Build the verification infrastructure:

1. Curate fixture set — 3–5 Electric Sheep flames, balanced across variation usage + visual
   style. Probably: 1 simple (244 era), 1 variation-heavy (247 era), 1 xaos-heavy (248 era),
   plus 1–2 from kotlin's `parity/goldens/` curated set.
2. Pre-render each with flam3-C; commit goldens to `fixtures/flam3-goldens/<id>/golden.png`.
3. Port kotlin's R formula from
   `parity/src/main/kotlin/pyr3/parity/Compare.kt` to `src/parity-r.ts`.
4. BE parity test path — Vitest suite invokes `pyr3-render.ts`, R-compares output.
5. FE parity test path — Vitest + `chrome-devtools-mcp` drives dev server, captures canvas,
   R-compares output.
6. Calibrate R tolerance threshold per fixture (kotlin's baseline R ≈ 1.5–3.0).

**Acceptance:** test rig produces R score for any (flame, consumer) pair; baseline R scores
captured for current code; threshold(s) chosen.

### Phase 3 — Iterate to v1.0 ship gate (target: v0.7 → v1.0)

Use the test rig to drive bug-fixing, shader improvements, parameter tuning until **both FE
and BE pass R tolerance for the entire fixture set**.

Each iteration is a discrete versioned ship (v0.7, v0.8, ...). v1.0 = the version that
satisfies the ship gate.

Per CLAUDE.md gameplay-tuning-sacrosanct: mechanism fixes (clamps, lifecycle, event wiring)
ship without asking; literal numeric tuning (damage / spawn-rate-equivalent constants like
filter radii, DE thresholds) require explicit user ask. Diagnose mechanism first.

**Acceptance:** v1.0 ship gate met. Trigger pulled for replacing
`MattAltermatt/pyr3` + `pyr3-peek` on GitHub.
