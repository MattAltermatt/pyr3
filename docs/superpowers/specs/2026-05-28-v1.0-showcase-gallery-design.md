# pyr3 v1.0 — `/showcase` public gallery (Chunk 1)

**Status:** locked (brainstorm 2026-05-28) · **Milestone:** v1.0 · **BACKLOG:** `[PYR3-007]` (Chunk 1 slice)

## Context

pyr3 is at the v1.0 doorstep — both parity ship-gates are green (v0.20). The
remaining v1.0 work is the public-facing surface plus the GitHub repo replacement.
That work was decomposed this session into four chunks; **this spec covers Chunk 1
only**: the static public showcase gallery at `/showcase`.

The repo already holds 55 pre-rendered 4K PNGs (`fixtures/showcase-v1.0/*.png`,
~928MB, gitignored) plus a committed `_manifest.json`, but nothing presents them.
The showcase is how visitors see what pyr3 produces.

### Scope decisions locked this session (supersede earlier `[PYR3-007]` notes)

- **Root `/` = the FE viewer** (front door, unchanged). **`/showcase` = the gallery.**
  This *reverses* the original "locked decision #1" that the unversioned root would BE
  the showcase — re-decided once the FE viewer's continued presence at root was recalled.
- **No gallery→viewer click-to-load in v1.0.** Deferred to the post-v1 sharing-mechanics
  chunk; `[PYR3-020]` (share-link >6KB bug) drops out of this chunk entirely.
- **4K images live ONLY on `gh-pages`, never in `main`.** Mirrors pyr3-kotlin's showcase
  (gh-pages branch held ~90% JPEGs + thumbnails; `main` stayed clean). Confirmed by
  inspecting `pyr3-kotlin` `origin/gh-pages`: per-fixture `<id>.gpu.4k.jpg` (~2–4MB) +
  `<id>.gpu.4k.thumb.jpg` (~90–310KB).
- **Unversioned URL** (`/showcase`, always-latest) — pyr3's intentional divergence from
  kotlin's `/v1.0/`, `/v1.1/` museum dirs.
- **PYR3-031 (FE viewer cleanup)** is now its OWN chunk (Chunk 3), NOT bundled here.

### The four chunks (this spec = Chunk 1)

## Design (reuse the pyr3-kotlin showcase format, with pyr3 deltas)

### Page chrome
- **Hero:** `▲ pyr3` wordmark + tagline ("A modern fractal-flame renderer in
  TypeScript + WebGPU") + ~50-word lineage lede (flam3 → flam3-kotlin → pyr3) +
  a link to the live viewer.
- **Nav:** github · CHANGELOG · VISION · live viewer →
- **Banner:** `pyr3 v<ver> · rendered <date> · <hardware> · 55 flames in ~9 min`.

### Per-card (masonry columns — true aspect, no cropping)
- Thumbnail wrapped in `<a href="./<id>.4k.jpg" target="_blank">` — the full 4K opens
  in a new tab (kotlin's proven behavior; **no** viewer link in v1).
- Id row: a persistent permalink `#` anchor + `<id>` (monospace) on the left, and an
  explicit **⤢ Open 4K** button right-aligned on the same line, close to the image (the
  obvious click affordance — mobile-safe, doesn't rely on hover; chose option C from the
  affordance brainstorm). The thumbnail itself stays clickable too.
- Attribution: "By **&lt;nick&gt;**" — `nick` from the source `.flam3` `nick=` attribute;
  graceful "artist unknown" fallback. (The "from electric-sheep-fold" suffix is deferred.)
- Render line: plain text "Rendered at `<W>×<H>` by pyr3 GPU in `<N.N>s`" — **time only,
  NO "×faster" / CPU-equivalent comparison** (locked decision #2).
- A mean-luminance gate (computed off the thumb via `jpeg-js`) auto-skips effectively-black
  renders so the gallery never shows an empty card (see `[PYR3-033]`/`[PYR3-034]`).
- Mobile: single column below 760px.
- **Deferred for now:** `.flame` per-fixture download (and its copy step) + the
  "from electric-sheep-fold" attribution suffix.

### Image tiers (both JPEG ~q90, from the gitignored 4K PNGs)
- `<id>.4k.jpg` — full-res (3840 long-edge), ~2–4MB. The click-through target.
- `<id>.thumb.jpg` — 600px long-edge, ~150KB. The grid image.
- All-55 footprint ≈ 110–170MB — fine for a gh-pages deploy artifact, never in `main`.

Format is **JPEG** (universal) over WebP, per user preference, even though WebP q82
measured ~30% smaller.

## Implementation

### `scripts/build-showcase.mjs` (new)
Single Node script (matches existing `scripts/*.mjs`; reuses the manifest-iteration
pattern from `scripts/build-showcase-v1.0-gallery.mjs`). Reads
`fixtures/showcase-v1.0/_manifest.json`
(`{generatedAt, totalSec, fixtures:[{id, source, renderSec}]}`). For each fixture, from
`fixtures/showcase-v1.0/<id>.pyr3-4k.png`:
- `sips -s format jpeg -s formatOptions 90` → `<id>.4k.jpg`.
- `sips -Z 600 -s format jpeg -s formatOptions 90` → `<id>.thumb.jpg`.
- Copy the source `.flame` → `<id>.flame`.
- Render dimensions from the PNG header (`pngjs`, already a dependency) for the pill.
- `nick` via regex on `<flame ... nick="...">` in the source `.flam3`.

Then generate a complete static `index.html` with all 55 cards inlined (dark theme
matching `index.html`'s `:root`: `--bg:#0a0a0c`, `--accent:#ff8c1a`). **All asset paths
relative** (`./<id>.thumb.jpg`) so the page is gh-pages base-path agnostic; viewer link
is `../`; github/CHANGELOG/VISION are absolute github.com URLs. Fixtures sorted by id
for stable output.

**Output dir: `public/showcase/`** — Vite serves `public/` at root in dev and copies it
to `dist/` on build, so one location satisfies both local preview (`/showcase/`) and the
deploy artifact with zero extra wiring.

Banner data (hardware/date/version): args/env on the script (`--hw "Apple M-series"`),
date defaulting to the manifest's `generatedAt`, version from `package.json`.

npm script: `"build:showcase": "node scripts/build-showcase.mjs"`.

### `.gitignore`
Add `public/showcase/` — the generated HTML is cheap but the JPEGs/.flames are heavy and
fully regenerable. Enforces the "never in main" invariant.

   55×(`.4k.jpg`,`.thumb.jpg`,`.flame`); spot-check a 4K JPEG ~2–4MB, a thumb ~150KB.

   unknown" fallback; permalink `#<id>` anchors; <760px → single column; console clean.
3. `npm run build` → `dist/showcase/` populated (deploy-artifact proof).
4. `git status` → `public/showcase/` ignored; nothing heavy staged.
5. `npm run typecheck && npm test` → no regressions (4494/4499 baseline).
