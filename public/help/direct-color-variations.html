Direct-color variations

A class of variations that color a flame xform from spatial position instead of the palette.

What palette-indexed coloring does

Every standard flam3 variation (the 99 in pyr3's core set — linear, spherical, julia, and so on) lets the chaos game color each scatter by sampling the genome's 256-stop palette. Each xform has a color value that picks where it sits on that palette, and a color_speed value that controls how fast the iterated color tracker is pulled toward it. Colors emerge from the palette gradient — the artist designs the palette, the genome chooses slices of it.

What direct-color (DC) variations do

Direct-color variations bypass the palette entirely. They compute RGB directly from the scatter's spatial position, then write that color into the histogram instead of the palette lookup. The result is a class of flames that palette-indexed coloring simply cannot make — marbled noise fields, discrete tiled regions, color that flows with geometry rather than with the xform's firing history.

DC variations originated in Neil Slater's Apophysis 7X plugin pack and became signature features of the JWildfire user community. pyr3's first wave ports four of the family.

The four DC variations in pyr3

DC dc_linear

Color from spatial coord, mapped through a clamped affine to RGB. The simplest possible DC variation — proves the override path with a one-line formula. No parameters.

DC dc_perlin

Color from a 2D Perlin noise field. The hue cycles with the noise value; saturation is full, lightness sits at 0.55 so colors read clearly against a dark background. Parameters: scale (noise frequency), octaves (fractional-Brownian-motion depth, 1–8), and color_seed (rotates the hue cycle — two dc_perlin xforms with different seeds produce different palettes from the same field).

This is the marquee DC look. The "marbled / painterly" texture in most JWildfire-era gallery flames comes from one or more dc_perlin xforms layered over a chaos-game attractor.

DC dc_gridout

Color by canvas quadrant. Floor the scatter position into integer cells (cells parameter controls density), then hash each cell to an RGB triple. Produces a tile / pixelated look, completely different visual register from the smooth perlin field.

DC dc_cylinder

The DC analog of flam3's cylinder — same position warp ((sin(x), y)) plus a color derived from the warped coordinates. Demonstrates the "DC + shape" pattern: a single variation contributing both a position change AND a color override. The hue spirals along x, lightness modulates with y.

How to use them

In the editor, open any xform's variation picker — the four DC variations live in their own Direct color category and each carries a DC badge. Add one to a chain like any other variation; the chain's color / color_speed rows annotate "(overridden by dc_*)" to signal that the palette lookup is bypassed for that xform.

Override semantics

When an xform's variation chain contains any DC variation, every scatter from that xform writes DC RGB instead of palette[color_index] at histogram time. The position contribution from each variation in the chain still sums normally — DC variations affect color only (the cylinder one is the exception: it warps position too). If a chain contains multiple DC variations, the last one in the chain wins for the color.

Existing genomes are unaffected. None of the 99 standard flam3 variations are DC, so any flame from before this feature renders bit-for-bit identically. The full parity test rig (26 flames rendered with pyr3 vs flam3-C) passes byte-identically with this feature live.
