Why does this take so long?

The “Why so long? ↗” link in the 4K render progress bar lands here. Short version: fractal flames are made of millions of tiny dot placements, and 4K means many more dots per pixel than the quick preview can afford.

What's actually happening

A fractal flame is an iterated function system (IFS). Roughly:

  1. Start with a random point in the plane.
  2. Pick one of the flame's “xforms” at random (each xform has a weight).
  3. Apply that xform — it's a math function that warps the point's position and assigns it a color.
  4. Splat a colored dot at the new position.
  5. Go back to step 2. Forever. (Or until enough dots have been splatted.)

This is called the chaos game. After enough iterations, the dots form a recognizable image — the fractal flame — even though each individual dot is “random.” 

          ┌────────┐
          │  pick  │ ──► xform 1 ──┐
          │ xform  │ ──► xform 2 ──┤
          │ (rand) │ ──► xform 3 ──┤
          └────────┘     ...       │
                                   ▼
                            ┌──────────────┐
                            │ apply, splat │ ──► one colored dot
                            │ a dot, then  │
                            │ loop forever │
                            └──────────────┘

Why “many dots per pixel” matters

Each pixel of the final image is the accumulation of all the dots that landed inside it, color-blended and tone-mapped. Few dots per pixel → noisy result (you see speckles where dots happened to land vs not). Many dots per pixel → smooth result (the chaos game settles into the true density of the flame).

The metric here is samples per pixel (SPP):

  • SPP = 4 → speckled, unrecognizable as a finished render
  • SPP = 16 → noticeably noisy but the shape is clear (what pyr3's quick preview uses by default)
  • SPP = 100 → smooth, gallery-ready
  • SPP = 1000+ → flam3's offline-render preset; effectively settled

Why 4K is much more than 4× the work of 1K

A 1024×1024 quick preview at SPP=16 needs:

1024 × 1024 × 16 = ~16,000,000 dots

A 4K render (4096×4096) at the SAME SPP needs:

4096 × 4096 × 16 = ~268,000,000 dots

That's 16× more dots just for the resolution jump — because the pixel grid grew by 4× horizontally AND 4× vertically (4² = 16).

And SPP=16 is the quick-preview floor. A nice 4K render typically wants SPP=50 or higher to hide the noise that's now spread across many more pixels — pushing the dot count past a billion.

Plus the spatial filter (anti-aliasing) and density estimator passes work on a buffer 16× larger, so those steps slow down too.

Why pyr3 chunks the dispatch

Submitting a billion-dot compute pass to the GPU as a single dispatch would lock the browser for the full duration. macOS Metal forcibly restarts the GPU process if anything takes longer than ~2 seconds (“TDR” — timeout detection and recovery).

pyr3's renderer breaks the work into chunks (~1 million dots each). Between chunks, the browser gets a frame to composite, the progress bar updates, and the cancel button stays responsive. The total work is identical — it just doesn't lock you out while it runs.

Cancel is safe

Hit ✕ Cancel mid-render and the in-flight chunk's GPU work still completes (the browser doesn't expose a way to abort GPU dispatches mid-flight), but no further chunks get queued. The progress bar disappears, the canvas keeps whatever partial result the last chunk produced, and the bar's buttons re-enable. Nothing leaks.

TL;DR

Quick (1024², SPP=16):  ~16M dots,   ~half a second
4K     (4096², SPP=50): ~800M dots,  many seconds (browser stays alive)

Same algorithm, same GPU, just more dots.

For the curious: the algorithmic details (palette interpolation, log-density tone map, ~Gaussian spatial filter, density estimator) all live in the source tree. Start with src/shaders/chaos.wgsl.
